前言

&nbsp;&nbsp;异步背后的“靠山”就是Event Loop。这里的异步准确的说叫浏览器的Event Loop或者JavaScript运行环境的Event Loop，因为ECMAScript中没有Event Loop，Event Loop是在HTML Standard定义的。
&nbsp;&nbsp;Event Loop是浏览器或Node解决单线程运行时不会阻塞的一种机制。规范中定义了浏览器何时进行渲染更新，了解它有助于性能优化.

定义

Event Loop

即事件循环，实现异步的一种方式。在HTML Standard中的定义章节：

为了协调事件，用户交互，脚本，渲染，网络等，用户代理必须使用本节所述的event loop。

触发一个click事件，进行一次ajax请求，背后都有event loop在运作。

Task（macrotask， 宏任务）

一个Event Loop有一个或者多个task队列
当用户代理安排一个任务，必须将该任务增加到相应的Event Loop的一个task队列中。
每一个task都来源于指定的任务源，比如可以为鼠标、键盘事件提供一个task队列，其他时间又是一个单独的队列。可以为鼠标，键盘事件分配更多的时间，保证交互的流畅。

规范在Generic task sources中有提及哪些是task任务源：

  1. DOM操作任务源，用于相应dom操作，例如一个元素以非阻塞的方式插入文档。
  2. 用户交互任务源，用于对用户交互作出反应，例如键盘或鼠标输入。
  3. 网络任务源：用于相应网络活动。
  4. history traversal 任务源，当调用history.back()等类似的api时，将任务插入task队列

总结，task任务源有哪些：

  • setTimeout
  • setInterval
  • setImmediate
  • I/O
  • UI rendering

microtask（微任务）

&nbsp;&nbsp;每个Event Loop都有一个microtask队列，microtask会进入microtask队列，而不是task队列

&nbsp;&nbsp;如果在初期执行时，spin the event loop，microtasks有可能被移动到常规的task队列，在这种情况下，microtasks任务源会被task任务源所用。通常情况，task任务源和microtasks是不相关的。

microtask 和 macrotask队列
&nbsp;&nbsp;相似之处：都是先进先出的队列，由指定任务源去提供任务。
&nbsp;&nbsp;不同的是，一个Event Loop里只有一个microtask队列。

HTML Standard没有具体指明哪些是microtask任务源，通常认为是microtask任务源有：

  • process.nextTick
  • promises
  • Object.observe
  • MutationObserver

Notes：
&nbsp;&nbsp;在Promises/A+规范的Notes 3.1中提及了Promise的then方法可以采用macrotask 或者 microtask机制来实现。因此会产生promise在不同浏览器的差异。当然现在的共识都是 promise 都属于 microtask 队列。

深入了解

&nbsp;&nbsp;有两种Event Loop，一种在浏览器上文，一种在workers中。
&nbsp;&nbsp;每一个用户代理必须至少有一个浏览器上下文Event Loop，但是每个单元的相似源浏览器上下文至多有一个Event Loop。
&nbsp;&nbsp;Event Loop总是具有至少一个浏览器上下文，当一个Event Loop的浏览器上下文全都销毁，Event Loop也会销毁。一个浏览器上下文总有一个Event Loop去协调它的活动。
&nbsp;&nbsp;Worker的Event Loop相对简单一些，一个Worker对应一个Event Loop，Worker进程模型管理Event Loop的生命周期

&nbsp;&nbsp;浏览器上下文是一个将Document对象呈现给用户的环境。在一个Web浏览器内，一个标签页或窗口常包含一个浏览上下文，如一个iframe或一个frameset内的若干frame。

总结：
&nbsp;&nbsp;每个线程都有自己的Event Loop。
&nbsp;&nbsp;浏览器可以有多个Event Loop，Browsing Contexts和Web Workers就是相互独立的.
&nbsp;&nbsp;所有同源的Browsing Contexts可以共用Event Loop，这样它们就可以互相通信。

Event Loop的循环过程

在规范的Processing model定义了Event Loop的循环过程

  1. 在tasks队列中选择最老的一个task，用户代理可以选择任何task队列，如果没有可选的任务，则跳到下边的microtask步骤。
  2. 将上边选择的task设置为正在运行的task。
  3. Run：运行被选择的task。
  4. 将Event Loop的currently running task 变为null。
  5. 从task队列里移除前边运行的task。
  6. 执行microtasks任务检查点。
  7. 更新渲染。
  8. 如果这是一个Worker EventLoop，但是没有任务在task队列中，并且WorkerGlobalScope对象的closing标识为true，则销毁event loop，中止这些步骤，然后进行定义在Web workers章节的run a worker。
  9. 返回第一步。

notes:
&nbsp;&nbsp;执行完microtask队列里的任务，有可能会渲染更新。浏览器很聪明，在一帧以内的多次dom变动浏览器不会立即响应，而是会积攒变动以最高60HZ的频率更新视图。

microtask检查点

&nbsp;&nbsp;以下为event loop运行第6步的详细步骤。

&nbsp;&nbsp;当用户代理去执行一个microtask checkpoint，如果microtask checkpoint的flag为false，用户代理必须运行下面的步骤：

  1. 将microtask checkpoint的flag设为true。
  2. Microtask queue handling: 如果Event Loop的microtask为空，直接跳到第八步。
  3. 在microtask队列中选择最老的一个任务。
  4. 将上一步选择的任务设为Event Loop的currently running task
  5. 运行选择的任务。
  6. 将Event Loop的currently running task 变为null。
  7. 将前面运行的microtask从microtask队列中删除，然后返回第二步。
  8. Done： 每一个environment settings object它们的 responsible event loop就是当前的event loop，会给environment settings object发一个 rejected promises 的通知。
  9. 清理IndexedDB的事务。
  10. 将microtask checkpoint的flag设为false。

什么时候会调用microtask checkpoint呢？

  • 当上下文执行栈为空，执行一个microtask checkpoint。
  • 在Event Loop的第六步执行checkpoint，也就是在运行task之后，更新渲染之前。

执行栈

&nbsp;&nbsp;task和microtask都是推入栈中执行的，要完整了解event loops还需要认识JavaScript execution context stack，它的规范位于https://tc39.github.io/ecma262/#execution-context-stack。

&nbsp;&nbsp;javaScript是单线程，也就是说只有一个主线程，主线程有一个栈，每一个函数执行的时候，都会生成新的execution context（执行上下文），执行上下文会包含一些当前函数的参数、局部变量之类的信息，它会被推入栈中， running execution context（正在执行的上下文）始终处于栈的顶部。当函数执行完后，它的执行上下文会从栈弹出。

完整异步过程

打个比喻：
&nbsp;&nbsp;主线程类似一个加工厂，只有一个条流水线（单线程），待执行的任务为原料，只有前一个加工完，才能进行后一个。eventLoop就是把原料放上流水线的工人。
&nbsp;&nbsp;同步任务：只要已经放在流水线上的，它们会被依次处理。
&nbsp;&nbsp;异步任务：待处理的原料，工人会按照它们的种类排序，在适当的时机放上流水线。

过程图：

举个栗子： 

Promise.resolve().then(function promise1() {  
  console.log('promise1');  
})  

setTimeout(function() {  
  console.log('setTimeout1');  

  Promise.resolve().then(function promise2() {  
    console.log('promise2');  
  })  
}, 0)  

setTimeout(function setTimeout2() {  
  console.log('setTimeout2')  
}, 0)

运行过程解析：
第一步：
&nbsp;task队列: [scipt]，microtask队列: []

&nbsp;script里的代码被列为一个task，放入task队列。

第二步：开始循环
&nbsp;循环1：
&nbsp;&nbsp;&nbsp;1. 从task队列中取出script任务，推入栈中执行。promise1列为microtask，setTimeout1列为task，setTimeout2列为task。
&nbsp;&nbsp;&nbsp;task队列：[setTimeout1 setTimeout2], microtask队列：[promise1]
&nbsp;&nbsp;&nbsp;2. script任务执行完毕，执行microtask checkpoint，取出microtask队列的promise1执行
&nbsp;&nbsp;&nbsp;task队列：[setTimeout1 setTimeout2], microtask队列：[]

&nbsp;循环2：
&nbsp;&nbsp;&nbsp;1. 从task队列中取出setTimeout1，推入栈中执行，将promise2列为microtask。
&nbsp;&nbsp;&nbsp;task队列：[setTimeout2], microtask队列：[promise2]
&nbsp;&nbsp;&nbsp;2. 执行microtask checkpoint，取出microtask队列的promise2执行。
&nbsp;&nbsp;&nbsp;task队列：[setTimeout2], microtask队列：[]

&nbsp;循环3：
&nbsp;&nbsp;&nbsp;1.从task队列中取出setTimeout2，推入栈中执行。
&nbsp;&nbsp;&nbsp;task队列：[], microtask队列：[]
&nbsp;&nbsp;&nbsp;2.setTimeout2任务执行完毕，执行microtask checkpoint。
&nbsp;&nbsp;&nbsp;task队列：[], microtask队列：[]
